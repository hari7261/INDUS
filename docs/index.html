Core Features

Everything you need. Nothing you don't.

All built around the native ind <command> [options] contract with a fast Go runtime underneath.

58 Native INDUS Commands

System, project, environment, filesystem, network, developer, package, terminal, workspace, and core commands all resolve under one native command surface.

ind sys stats   ind env list   ind proj create

Registry-Backed Resolution

Commands resolve through core/commands/registry.json, giving fast lookup, scalable metadata, category ownership, and version tracking.

core/commands/registry.json

Fast Execution Engine

Lazy module loading, command caching, minimal spawning, and async-safe execution keep local commands fast and predictable.

ind sys stats   ind term speed

Project and Workspace Flows

Create, initialize, build, run, archive, and switch workspaces without colliding with other terminal ecosystems.

ind proj create orbit-app --dir .

Versioned Documentation UI

The landing page now connects to dedicated version pages, command reference pages, and release history without losing the original INDUS visual identity.

docs/commands.html   docs/versions.html

SEO and AI Metadata

Structured schema, OpenGraph tags, and robots metadata make the command model readable to search engines and AI tooling.

Quick Start

Start building immediately

Native commands, registry-backed resolution, and versioned docs from the same landing page.

System & Network

INDUS > ind sys stats
INDUS > ind net status --url https://example.com
INDUS > ind net pingx example.com --port 443
INDUS > ind net fetch https://example.com

Project & Workspace

INDUS > ind proj create orbit-app --dir .

INDUS > ind work init orbit-space

Environment & Files

Configuration

INDUS reads %USERPROFILE%\.config\indus\config.yaml on startup.

~/.config/indus/config.yaml 
api_timeout = 30    # HTTP timeout in seconds
max_retries = 3     # Retry count for idempotent methods

Override path: set INDUS_CONFIG=C:\path\to\config.yaml

Command Engine

Registry-backed native command execution

The v1.4.1 runtime parses native ind commands, resolves them through the registry, and executes only the module it needs.

Input
ind sys stats
↓ parseCommandLine() + registry lookup
Stage 1 — parser
tokens → command path → parsed options
lightweight parsing without a heavy CLI framework
registry.json
Stage 2 — lazy module
system.Execute(ctx, invocation)
cached reads, safe wrappers, and structured responses
Panic recovery prevents a single command from crashing the terminal
Short-lived read commands can return cached results for rapid local execution
Unknown or invalid input returns structured IND error codes with safe suggestions
Module loading is lazy, so only the active command category enters memory
